# Tech Digest — 2026-07-05

**1. Cloudflare sets deadline to block AI crawlers that bundle search with AI training**
Cloudflare is pushing AI crawlers to separate search indexing from model-training access, tightening the default economics of web-scale data collection. Engineering leaders should expect more bot policy, content licensing, and crawler-identification work to become infrastructure concerns, not just legal ones. [Read](https://www.nbcnews.com/tech/tech-news/cloudflare-sets-ai-crawler-deadline-separate-search-blocked-rcna352446)

**2. Better Models: Worse Tools**
Armin Ronacher documents a regression where newer Claude models behave worse with tool calls despite stronger raw model capabilities. The takeaway for agent teams is practical: benchmark the whole tool-using system, because model upgrades can silently break workflows that previously worked. [Read](https://lucumr.pocoo.org/2026/7/4/better-models-worse-tools/)

**3. Bad Epoll (CVE-2026-46242)**
This writeup/repo highlights a Linux `epoll` vulnerability, a reminder that low-level eventing primitives remain part of the security boundary for high-scale services. Infra teams running dense multi-tenant Linux workloads should track kernel fixes and threat models around async I/O paths. [Read](https://github.com/J-jaeyoung/bad-epoll)

**4. A new trend: smart model routing**
Smart model routing is becoming a key pattern for balancing cost, latency, and quality across multiple LLMs instead of betting on a single frontier model. For engineering leaders, this points to an AI platform layer with evaluation, routing policy, fallback behavior, and spend controls as first-class capabilities. [Read](https://blog.pragmaticengineer.com/the-pulse-a-new-trend-smart-model-routing/)

**5. Does Code Cleanliness Affect Coding Agents?**
A controlled study asks whether cleaner codebases materially improve coding-agent performance. As teams roll agents into development workflows, code health may become not just a human-maintainability issue but a lever for agent reliability and throughput. [Read](https://arxiv.org/abs/2605.20049)
