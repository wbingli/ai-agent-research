# Tech Digest — 2026-05-10

**1. Local AI needs to be the norm**
A strong argument that sensitive AI workloads should move toward local-first execution instead of defaulting to cloud APIs, especially as models become good enough for many personal and enterprise tasks. For engineering leaders, the takeaway is architectural: privacy, latency, cost, and resilience may increasingly favor hybrid/local inference paths. [Read](https://unix.foo/posts/local-ai-needs-to-be-norm/)

**2. Hardware Attestation as Monopoly Enabler**
GrapheneOS warns that device attestation can shift from a security primitive into a gatekeeping mechanism that lets platform owners decide which OSes, apps, and users are “trusted.” This matters for anyone building products on mobile or edge platforms because security controls can become ecosystem lock-in and distribution risk. [Read](https://grapheneos.social/@GrapheneOS/116550899908879585)

**3. Obsidian plugin was abused to deploy a remote access trojan**
A malicious Obsidian plugin campaign shows how extensible productivity tools can become supply-chain attack surfaces, especially when plugins get broad filesystem access and users trust community marketplaces. Engineering orgs should treat editor, note-taking, and AI-assistant extensions like code dependencies: reviewed, pinned, monitored, and revocable. [Read](https://cyber.netsecops.io/articles/obsidian-plugin-abused-in-campaign-to-deploy-phantom-pulse-rat/)

**4. WebRTC is the Problem**
This Media over QUIC post argues that WebRTC is an awkward fit for modern realtime AI voice systems, where developers want simpler transport, lower operational complexity, and better control over media behavior. It is a useful lens for teams designing low-latency voice/video or agent streaming infrastructure: protocol choices become product velocity choices. [Read](https://moq.dev/blog/webrtc-is-the-problem/)

**5. Idempotency Is Easy Until the Second Request Is Different**
A practical breakdown of the hard parts of idempotent APIs: mismatched retries, concurrent submissions, partial downstream success, response replay, expiry, and duplicate message handling. It is a good reminder that reliable distributed systems require explicit semantics, not just an `Idempotency-Key` header. [Read](https://blog.dochia.dev/blog/idempotency/)
