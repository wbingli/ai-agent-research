# Tech Digest — 2026-05-07

**1. AlphaEvolve: Gemini-powered coding agent scaling impact across fields**
DeepMind describes AlphaEvolve, a Gemini-based coding agent used to discover and optimize algorithms across genomics, quantum computing, data-center scheduling, and other infrastructure-scale problems. The important signal is that agents are moving from code generation demos toward measurable optimization loops in scientific and production systems. [Read](https://deepmind.google/blog/alphaevolve-impact/)

**2. Dirtyfrag: Universal Linux LPE**
A new Linux local privilege-escalation issue, “Dirty Frag,” is being discussed on oss-security as a broadly applicable kernel vulnerability class. Engineering leaders should treat this as a patch-prioritization and fleet-exposure problem: kernel bugs with universal LPE potential can quickly turn any foothold into root. [Read](https://www.openwall.com/lists/oss-security/2026/05/07/8)

**3. Agents need control flow, not more prompts**
This piece argues that reliable agents need explicit software control flow—state machines, retries, branching, and invariants—rather than ever-longer prompt chains. It is a useful framing for teams trying to productionize agents: the durable engineering work is orchestration, observability, and failure handling. [Read](https://bsuh.bearblog.dev/agents-need-control-flow/)

**4. DeepSeek 4 Flash local inference engine for Metal**
Salvatore Sanfilippo released `ds4`, a local inference engine targeting Apple Metal for DeepSeek 4 Flash. Local, device-optimized inference keeps getting more practical, which matters for latency, privacy, cost control, and products that cannot depend on round trips to hosted model APIs. [Read](https://github.com/antirez/ds4)

**5. Potential Consequences of Using Postgres as a Job Queue**
This writeup reviews the hidden costs of using Postgres as a queue, especially around table churn, locking, vacuum pressure, and performance cliffs. It is a good reminder that “reuse the database” is often right early on, but queue workloads need explicit operational limits before they become reliability incidents. [Read](http://richyen.com/postgres/2026/05/04/postgres_job_queue.html)
